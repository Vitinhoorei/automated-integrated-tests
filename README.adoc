== 🇧🇷 Versão em Português

=== 📝 Descrição

O *SAP AI-Driven Automation Tester* é uma solução desenvolvida em *Python* para automatizar testes unitários e integrados no sistema *SAP*, combinando automação de interface, processamento com Inteligência Artificial, execução orientada por planilhas Excel e captura estruturada de evidências.

O sistema nasceu com foco no módulo *PM (Plant Maintenance)*, mas vem evoluindo para uma *Engine Core Agnóstica de Módulos SAP*, preparada para suportar diferentes fluxos e módulos, como *PM, PP, MM e SD*, sem a necessidade de reescrever o núcleo da aplicação.

As regras específicas de cada módulo são tratadas como expansões da solução, permitindo escalabilidade, manutenção simplificada e reaproveitamento da base técnica.

---

=== 🎯 Problema e Impacto

Tradicionalmente, a validação de fluxos no SAP exige alto esforço manual, tanto na execução quanto na coleta de evidências.

*Cenário anterior*

* Execução manual de testes
* Captura manual de evidências
* Alto risco de erro humano
* Maior tempo operacional para validação dos fluxos

O ciclo completo de testes levava *semanas* para ser concluído.

*Com a automação*

O sistema executa os cenários automaticamente, registra evidências e reduz a dependência de atividades repetitivas e manuais.

Hoje, a execução pode ser concluída em *alguns dias*, com muito mais consistência e rastreabilidade.

*Resultado*

Redução superior a *95% no tempo operacional*, além de:

* padronização das evidências
* redução de erros humanos
* aumento da rastreabilidade dos testes
* melhoria da escalabilidade da operação

---

=== ⚙️ Funcionalidades Principais

* *Interpretação de Cenários via Excel*
  Leitura de massas de teste, regras de fluxo e parâmetros definidos em planilhas.

* *Automação SAP GUI*
  Execução nativa de TCODEs, preenchimento de campos, navegação entre telas e interação com elementos da interface via SAP GUI Scripting (*win32com*).

* *Memória de Contexto*
  Captura de IDs e dados dinâmicos gerados pelo SAP, como ordens e documentos, reutilizando essas informações em etapas posteriores do fluxo.

* *Captura de Evidências*
  Geração automática de prints, registros de execução e saídas estruturadas para auditoria e validação.

* *Monitoramento de Erros e Dumps*
  Identificação de falhas técnicas durante a execução, permitindo análise mais rápida dos problemas.

* *Processamento com IA*
  Uso de Inteligência Artificial para apoio em decisões de fluxo, interpretação de cenários e futura reexecução inteligente de testes com correção assistida.

=== 🚀 Evolução da Solução

A solução vem recebendo melhorias voltadas para flexibilidade, robustez e experiência de uso.

==== Novo modelo de execução de planilhas

O sistema passou a trabalhar com dois modos distintos:

* *Modo Simulação*
  Permite validar o fluxo sem executar de fato no SAP, evitando a necessidade de remover transações da planilha apenas para testes.

* *Modo Execução*
  Executa o fluxo completo no ambiente SAP.

==== Lembretes contextuais na planilha

Foram adicionados lembretes no estilo *post-it* nas colunas da planilha para indicar:

* o significado da coluna
* o tipo de dado esperado
* o objetivo do campo dentro do fluxo

Isso facilita o uso do sistema após longos períodos sem operação e reduz a curva de reaprendizado.

==== Geração inteligente de resultados

A saída da automação não sobrescreve mais a planilha de entrada.

Agora o sistema gera automaticamente arquivos com padrão de nomenclatura como:

*Resultado de Automação - PP*
*Resultado de Automação - PM*

Isso evita confusão entre arquivo de entrada e resultado processado, além de melhorar a rastreabilidade das execuções.

==== Horário dinâmico no apontamento PM

No apontamento de ordens de manutenção, o horário passou a ser calculado dinamicamente com a lógica:

*hora atual - 2 horas*

Esse ajuste foi implementado conforme necessidade do processo e mantém o fluxo automático após o preenchimento.

==== Datas dinâmicas no módulo PP

No fluxo *CO01*, a data de início passou a utilizar sempre a data atual do sistema, evitando manutenção manual do campo.

No fluxo *CO02* de modificação de datas, o sistema preenche apenas a nova data de início e deixa que o próprio SAP calcule automaticamente a data final após a confirmação.

Além disso, foi considerado o comportamento do calendário de fábrica. Como o SAP pode bloquear datas que caiam em sábado ou domingo, o sistema evita esse tipo de inconsistência e conduz o preenchimento de forma que o próprio SAP faça o cálculo da data final de maneira limpa e compatível com a regra de calendário.

==== Tratamento inteligente de operações na CO11N

Como uma ordem pode possuir múltiplas operações, o sistema passou a:

* abrir o popup de operações
* mapear todas as operações disponíveis
* capturar os códigos de cada operação
* executar o fluxo uma operação por vez

Com isso, evita-se execução com operação em branco, fictícia ou incompleta, aumentando a confiabilidade do cenário.

=== 🧠 Melhoria em Desenvolvimento: Reexecução Inteligente com IA

Uma evolução já iniciada no sistema é a capacidade de *reexecutar testes automaticamente com apoio de IA* quando um erro é identificado por falta ou inconsistência de dados.

A proposta é que, ao detectar uma falha, o sistema:

* identifique a linha da planilha correspondente ao teste
* analise os dados já preenchidos
* consulte um mini banco de dados interno com valores possíveis e válidos
* proponha a melhor correção
* refaça o teste automaticamente com o novo valor

==== Caso inicial implementado

O primeiro cenário iniciado para essa inteligência é o campo *TipoAtvMnt* (*Tipo de Atividade de Manutenção*).

Se o usuário deixar esse campo em branco na planilha e o teste falhar por esse motivo, a IA avalia o contexto do cenário e consulta o mini banco de dados interno para identificar o valor mais adequado.

Em um caso já iniciado, o sistema consegue inferir que o valor *31* é o mais adequado para aquele teste e reexecuta o fluxo com esse preenchimento.

==== Status atual

* Estrutura inicial criada
* Primeira regra funcional implementada
* Reexecução automática em cenário inicial disponível
* Arquitetura em evolução para suportar novos campos e regras

==== Próximos passos previstos

* expansão para outros campos obrigatórios
* evolução do mini banco de dados interno
* estudo de integração com API ou base de dados SAP, se aplicável
* registro de decisões automáticas em log
* amadurecimento do mecanismo de confiança da IA

Essa frente ainda está em desenvolvimento, mas já representa um avanço importante rumo a uma automação mais resiliente, adaptativa e inteligente.

== 🇺🇸 English Version

=== 📝 Description

The *SAP AI-Driven Automation Tester* is a *Python-based solution* built to automate unit and integrated tests in the *SAP* environment, combining interface automation, Artificial Intelligence, Excel-driven execution and structured evidence capture.

The project was originally focused on the *PM (Plant Maintenance)* module, but it is evolving into an *agnostic SAP Core Engine* capable of supporting multiple SAP modules such as *PM, PP, MM and SD* without rewriting the core foundation.

Module-specific rules are treated as extensions of the solution, which improves scalability, maintainability and long-term reuse of the technical base.

---

=== 🎯 Problem and Impact

Traditionally, validating SAP process flows requires significant manual effort for both execution and evidence collection.

*Previous scenario*

* Manual test execution
* Manual evidence capture
* High risk of human error
* Longer operational time for validation

A complete testing cycle used to take *weeks* to be completed.

*With automation*

The system executes scenarios automatically, records evidence and reduces dependency on repetitive manual activities.

Today, execution can be completed in *a few days*, with much more consistency and traceability.

*Result*

More than *95% reduction in operational time*, along with:

* standardized evidence
* reduced human errors
* improved test traceability
* better operational scalability

---

=== ⚙️ Key Features

* *Excel-Driven Scenario Interpretation*
  Reads test datasets, execution logic and flow definitions from spreadsheets.

* *SAP GUI Automation*
  Native execution of SAP TCODEs, field input, screen navigation and GUI interaction using SAP GUI Scripting (*win32com*).

* *Context Memory*
  Captures dynamic SAP-generated data, such as order numbers and document identifiers, and reuses them in later steps.

* *Evidence Capture*
  Automatically generates screenshots, execution logs and structured outputs for audit and validation purposes.

* *Error and Dump Monitoring*
  Identifies technical failures during execution, enabling faster troubleshooting.

* *AI-Assisted Processing*
  Uses Artificial Intelligence to support flow decisions, scenario interpretation and future intelligent retry capabilities with assisted correction.

=== 🚀 Solution Evolution

The solution has been receiving improvements focused on flexibility, robustness and usability.

==== New spreadsheet execution model

The system now supports two execution modes:

* *Simulation Mode*
  Allows validation of the flow without actually executing it in SAP, avoiding the need to remove transactions from the spreadsheet just to perform tests.

* *Execution Mode*
  Runs the full process in SAP.

==== Spreadsheet reminders

Contextual notes in a *post-it* style were added to spreadsheet columns to explain:

* what the column means
* which type of value is expected
* what role that field plays in the flow

This improves usability, especially when the system is used again after a long period.

==== Smart result generation

The automation output no longer overwrites the input spreadsheet.

Instead, the system automatically creates result files using names such as:

*Automation Result - PP*
*Automation Result - PM*

This improves traceability and avoids confusion between source and output files.

==== Dynamic time handling in PM confirmation

For maintenance order time confirmation, the system now applies the following logic dynamically:

*current time - 2 hours*

This was implemented to meet a process requirement while keeping the flow fully automated.

==== Dynamic dates in PP

In *CO01*, the start date now always uses the current system date, avoiding manual maintenance.

In *CO02* date changes, the system fills only the new start date and lets SAP automatically calculate the end date after confirmation.

In addition, factory calendar behavior was taken into account. Since SAP may block dates that fall on Saturdays or Sundays, the system avoids this type of inconsistency and guides the input so SAP can calculate the final date cleanly and according to calendar rules.

==== Intelligent operation handling in CO11N

Since an order may contain multiple operations, the system now:

* opens the operation popup
* maps all available operations
* captures each operation code
* executes the flow one operation at a time

This avoids blank, fictitious or incomplete operations and increases execution reliability.

=== 🧠 Improvement in Progress: Intelligent Retry with AI

One enhancement already started is the ability to *automatically retry failed tests with AI assistance* whenever an error occurs due to missing or inconsistent data.

The idea is that, when a failure is detected, the system will:

* identify the spreadsheet row related to the failed test
* analyze the data already provided
* consult an internal mini database with valid and possible values
* determine the best correction
* automatically rerun the test with the corrected value

==== Initial implemented case

The first scenario under development for this capability is the *TipoAtvMnt* field (*Maintenance Activity Type*).

If the user leaves this field blank in the spreadsheet and the test fails because of that, the AI analyzes the test context and checks the internal mini database to determine the most appropriate value.

In an initial working case, the system can infer that the value *31* is the best fit for that scenario and rerun the flow using that value.

==== Current status

* Initial structure created
* First functional rule implemented
* Automatic retry available in an initial scenario
* Architecture evolving to support new fields and rules

==== Planned next steps

* extend support to additional mandatory fields
* evolve the internal mini database
* evaluate integration with SAP APIs or SAP data sources, when applicable
* log automatic decisions
* improve AI confidence handling

This capability is still in an early stage, but it already represents an important step toward a more resilient, adaptive and intelligent automation platform.
